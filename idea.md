1. 为每个 agent 都创建一个自己的工作区，它们的所有产出文件都可以分门别类的放进去。
2. agents 在表述已经做过的事情的时候，必须只基于其工作区中的内容，而不能胡编乱造。
3. agents 之间应该有专门的交流通道，用来同步信息。也许可以做个类似聊天记录的东西，把每个 agent 的发言都记录下来。这个聊天记录可以作为交流通道，也方便用户检查。
4. agents 之间可以互相拜托做事情，但这些事情必须经过用户同意后才能进行。
5. 每个 agent 都维护自己的状态，这样当它们需要导师（玩家）、别的 agent 做什么的时候，或者它有什么阶段性成果可以汇报的时候，可以自主汇报。
6. 既然我们的 agents 都有自己的状态循环，那么“下一天”和组会这种设计就不是很有用了。或许可以结合上面说的通讯录，每个 agent 都和导师（玩家）有一对一的联系，agents 之间也可以拉群，但必须加上导师。这样，每个 agent 都可以自主地向导师或者别的 agent 发起对话，汇报自己的工作进展，提出自己的想法和请求。用户可以随时介入，参与到对话中，或者向任意一个 agent 发起对话。对话结束后，这个对话会被记录在通讯录中，方便用户以后查阅。
7. 每当通讯录中的信息有了更新（有新消息后），所有相关的 agents 都需要被重新唤醒，以更新自己的状态。在它们被唤醒后，会根据新的信息，判断自己接下来的行动，并将其记录到工作区中。而工作区的更新，又可能导致通讯录的信息发生变化，从而形成一个闭环。在用户界面，有新消息的通讯录频道应该有红点提醒，并置顶。
8. 对于经济系统，也许我们可以直接读取用户使用的 api key 余额（这可能是钱数，也可能是 token 数），然后将这个作为研究组的经费。当需要使用 api 的时候，就从这个余额中扣除相应的费用。我们现在用的 minimax token plan 似乎是可以请求余额的。另外，对于 token plan，是每过一段时间刷新一次余额，我们也要考虑到。
9. 我们可以将整个项目先定位成辅助 agent 研究的科研人员的工具/游戏。这样，我们可以直接用 env 中配置好的 api key 来调用大模型（并从经济系统中扣款，反应现实）。经济系统也计算每个 agent 的大模型请求。这样，这个项目的经济系统就应该实时的反馈玩家的真实余额（即，大模型使用额度）。
